# mini_NPU

## 1. 프로젝트 개요
본 프로젝트는 3x3 필터와 입력 패턴 간의 MAC(Multiply-Accumulate) 연산을 수행하고 유사도를 판별하는 미니 NPU(Neural Processing Unit) 시뮬레이터입니다.

## 2. 주요 기능 및 동작 검증
프로젝트에서 요구하는 두 가지 모드의 정상 동작을 검증합니다.

### 모드 1: 단일 패턴 검사
* **기본 동작:** 3x3 필터 2개와 3x3 패턴을 입력받아 MAC 점수, 최종 판정 결과, 소요 시간(ms)을 출력합니다.
* **동점 처리:** 허용 오차 내에서 동점이 발생할 경우 "판정 불가"를 정상적으로 출력합니다.
* **오류 처리:** 입력된 행/열의 개수가 불일치하는 등 형식 오류 발생 시, 프로그램이 종료되지 않고 재입력을 유도합니다.

### 모드 2: 대량 데이터 로드 및 검사
* **데이터 로드:** JSON 형태의 데이터를 성공적으로 로드하고, 올바른 필터를 매칭하여 선택합니다.
* **라벨 정규화:** `expected` 값과 `filter` 키의 형식을 통일하는 표준 라벨 정규화를 수행합니다.

## 4. 코드 구조 및 설계
* **MAC 연산 로직의 전체 흐름:**
  입력 데이터(패턴)와 필터 배열을 `mac_operation` 함수로 전달하여 각 요소별 곱셈 및 누산(MAC)을 수행합니다. 연산된 결과 점수들은 딕셔너리 형태로 묶여 `compare_scores` 함수로 전달되며, 이 중 최고 점수를 가진 필터 이름을 동적으로 추출하여 최종 판정을 내립니다.

* **패턴 크기 추출 및 매칭 로직 설계:**
  입력된 데이터의 키(예: `test_5_01`)에서 `split("_")`을 사용해 배열의 크기(N=5)를 동적으로 추출합니다. 이를 통해 `size_5`와 같은 필터 그룹 키를 생성하여, 하드코딩 없이 다양한 크기의 패턴에 맞는 필터들을 자동으로 매칭하도록 설계했습니다.

* **라벨 정규화 모듈 분리 이유 및 장점:**
  "Cross", "cross", "+", "X", "x" 등 데이터 포맷의 파편화로 인한 오답 처리를 방지하기 위해 `normalize_label` 함수를 분리했습니다. 이를 통해 핵심 연산 로직을 건드리지 않고도 데이터 전처리가 가능해져 코드의 가독성과 유지보수성이 크게 향상되었습니다.

## 5. 핵심 기술 원리 적용
* **MAC 점수와 유사도 판단의 관계:**
  MAC 연산은 수학적으로 두 행렬의 내적(Dot Product)과 같습니다. 필터의 가중치(1)가 있는 위치에 입력 패턴의 값(1)이 일치할수록 곱셈 결과가 누적되어 큰 값이 나옵니다. 따라서 MAC 점수가 높을수록 두 패턴의 공간적 형태가 유사하다고 판단할 수 있습니다.

* **부동소수점 오차 및 비교 정책:**
  파이썬의 실수 연산 과정에서 미세한 오차가 발생하여 동일한 점수가 다르게 취급되는 것을 막기 위해 `1e-9` 크기의 Epsilon(허용 오차)을 도입했습니다. 최고 점수와 비교 대상의 점수 차이가 `1e-9` 미만일 경우, 동일한 점수로 간주하여 안전하게 동점자 처리를 수행합니다.

* **시간 측정 시 I/O 배제 경계:**
  순수한 연산 성능만을 벤치마크하기 위해, 시간 측정 함수인 `time.perf_counter()`를 MAC 연산 반복문(`for`)의 직전과 직후에만 배치했습니다. 파일 로드, 변수 초기화, `print()` 출력 등의 I/O 작업은 측정 경계선 밖에 두어 측정값의 신뢰도를 높였습니다.

* **동점 처리 규칙의 차이:**
  모드 1은 사용자와의 상호작용이 목적이므로 가독성을 위해 "판정 불가"라는 문자열을 출력합니다. 반면, 모드 2는 자동화된 테스트 채점기이므로, 동점이 발생하면 즉시 `raise ValueError`를 발생시켜 "UNDECIDED"라는 명확한 시스템 로그와 함께 FAIL 카운트에 집계되도록 예외 처리 기반으로 분리 설계했습니다.

## 6. 심층 분석 및 확장성 (심층 인터뷰)
* **연산 횟수와 O(N²) 경향:**
  NxN 크기의 2차원 배열을 순회하며 원소별 곱셈을 수행하므로, 입력 데이터의 한 변의 길이(N)가 증가할 때마다 연산 횟수는 N의 제곱에 비례하여 증가합니다. (예: 3x3=9회, 25x25=625회). 따라서 알고리즘의 시간 복잡도는 O(N²)으로 수렴합니다.

* **새로운 라벨 추가 시 확장 순서:**
  현재 코드는 필터 개수에 구애받지 않는 동적 딕셔너리 구조로 설계되었습니다. 따라서 새로운 라벨을 추가하려면 1) `data.json`에 새 필터 배열을 추가하고, 2) `main.py`의 `normalize_label` 함수에 새 라벨의 예외 처리(대소문자 등) 규칙만 한 줄 추가하면 됩니다. 메인 루프 코드는 단 한 줄도 수정할 필요가 없습니다.

* **대규모 패턴(1000x1000) 처리 문제 및 최적화:**
  1000x1000 배열은 한 번의 판정에 1,000,000번의 이중 포문 연산을 요구하므로 파이썬의 순수 직렬 처리에서는 심각한 속도 저하가 발생합니다. 이를 최적화하기 위해서는 이중 리스트를 1차원으로 펼쳐 캐시 히트율을 높이거나, `NumPy`와 같은 C언어 기반의 벡터화(Vectorization) 라이브러리를 활용해 연산 속도를 끌어올려야 합니다.

* **직렬 및 병렬 처리 관점의 병목:**
  현재의 직렬 처리는 CPU의 단일 코어만 사용하므로 반복 횟수가 많아질수록 순수 연산 시간이 병목이 됩니다. 멀티프로세싱이나 GPU 연산을 도입하면 행렬의 각 행이나 원소를 동시에 계산하여 시간을 획기적으로 단축할 수 있습니다. 다만, 배열의 크기가 작을 때는 오히려 병렬 처리를 위해 데이터를 분배하고 취합하는 오버헤드(IPC, 메모리 복사 등)가 새로운 병목 지점이 될 수 있습니다.
